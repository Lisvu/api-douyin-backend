**标题：从用户进入到数据闭环**

推荐流程：

1. 用户注册/登录，获取 Token。
2. 进入推荐流，系统返回推荐视频。
3. 用户观看视频，后端写入观看历史。

### 第 5 页：RESTful API 设计规范

**标题：RESTful API 规范落地**

建议内容：

| 规范点 | 项目做法 | 示例 |
| --- | --- | --- |
| 统一版本前缀 | 所有业务接口使用 `/api/v1` | `/api/v1/videos` |
| 资源名 URI | 使用资源名，不使用动词路径 | `/api/v1/users/me/videos` |
| HTTP 方法表达动作 | GET 查、POST 建、PUT 改、DELETE 删 | `DELETE /api/v1/videos/{id}` |
| 查询参数 | 分页和筛选使用 query string | `?page=1&limit=6` |
| 状态码 | 参数错误、未登录、无权限、资源不存在分别返回对应状态码 | 400 / 401 / 403 / 404 |
| 统一响应结构 | 返回 `success`、`message`、`data` | 便于前端统一处理 |

讲述重点：

这一页是课程评分重点。可以强调原来容易出现动词式路径，后来统一调整成资源式路径，例如发布视频不是 `/api/videos/publish`，而是 `POST /api/v1/videos`。

### 第 6 页：核心接口示例

**标题：核心接口设计示例**

建议内容：

| 业务 | 方法与路径 | 说明 |
| --- | --- | --- |
| 注册 | `POST /api/v1/auth/register` | 创建用户 |
| 登录 | `POST /api/v1/auth/login` | 返回 JWT Token |
| 推荐视频 | `GET /api/v1/videos/recommendations` | 返回推荐流 |
| 记录观看 | `POST /api/v1/videos/{id}/views` | 写入观看历史 |
| 点赞/取消点赞 | `PUT /api/v1/videos/{id}/like` | 幂等更新点赞状态 |
| 发布视频 | `POST /api/v1/videos` | 上传视频并创建资源 |
| 我的作品 | `GET /api/v1/users/me/videos` | 查询当前用户发布的视频 |
| 删除视频 | `DELETE /api/v1/videos/{id}` | 删除本人视频 |
| AI 文案 | `POST /api/v1/ai/video-copies` | 生成标题、简介和话题 |
| 请求日志 | `GET /api/v1/admin/request-logs` | 管理员查看日志 |

讲述重点：

不要把所有接口念一遍。挑 3 个典型接口说明设计思想：

- `POST /api/v1/videos/{id}/views`：把观看行为建模为视频的子资源。
- `PUT /api/v1/videos/{id}/like`：点赞状态更新使用 PUT，避免重复点赞导致数据异常。
- `GET /api/v1/users/me/videos`：用 `me` 表达当前登录用户资源。

### 第 7 页：数据与安全设计

**标题：数据存储与权限控制**

建议内容：

| 数据表 | 支撑功能 |
| --- | --- |
| `users` | 用户、角色、头像、通知已读时间 |
| `videos` | 视频元数据、作者、标题、封面、点赞数 |
| `views` | 观看历史、推荐去重 |
| `likes` / `favorites` | 点赞、收藏、互动状态 |
| `comments` / `danmaku` | 评论、回复、弹幕 |
| `watch_later` | 稍后再看 |
| `user_relations` | 关注、粉丝、好友 |
| `chat_messages` / `shares` | 私信和视频分享 |
| `request_logs` | 请求日志和耗时统计 |

权限控制重点：

- 除注册和登录外，核心接口统一校验 JWT。
- 删除视频时校验当前用户是否为视频作者。
- 我的观看历史、我的视频、通知列表只能访问当前用户自己的数据。
- 管理端日志和统计接口仅管理员可访问。

讲述重点：

这一页突出后端底层能力：不是只把数据存进去，还做了身份隔离、越权拦截和日志留痕。

### 第 8 页：AI 辅助与扩展亮点

**标题：AI 辅助创作与互动增强**

建议内容：

| AI 能力 | 输入 | 输出 |
| --- | --- | --- |
| 视频文案生成 | 视频首帧、文件名 | 标题、简介、话题标签 |
| 视频分类推荐 | 首帧、标题、简介、文件名 | 分类、置信度、推荐原因 |
| 视频封面推荐 | 候选封面帧、标题/简介 | 推荐封面序号、排序理由、优化建议 |
| 评论回复建议 | 评论内容、视频上下文 | 可直接使用或微调的中文回复 |

容错设计：

- AI 服务未配置时返回 `503 Service Unavailable`。
- 外部 AI 调用失败时返回 `502 Bad Gateway` 或 `500`。
- AI 失败不阻断基础发布流程，用户仍可手动填写标题、分类和封面。

讲述重点：

AI 模块是项目亮点，但要讲得务实：它不是替代核心业务，而是增强发布体验和互动效率，并且做了失败兜底。

## 四、8 分钟讲稿简版

可以按下面节奏讲，不建议逐字背，但可以作为口径。

大家好，我们是 API 课程第四小组，我们的项目是一个轻量短视频社区。项目围绕短视频产品中最核心的业务链路展开，包括用户注册登录、推荐流浏览、观看去重、点赞评论收藏、视频发布、我的视频管理、社交关系、AI 辅助创作和后台监控。

我们选择短视频社区作为主题，是因为它天然包含资源建模、状态更新、权限控制、分页查询、文件上传、日志监控等典型 API 场景，比较适合体现 RESTful API 的课程要求。项目边界上，我们没有做商业级推荐模型、直播、电商和复杂审核，而是聚焦课程验收需要的核心闭环。

从业务流程看，用户登录后进入推荐流，系统按照点赞热度和观看历史去重返回视频；用户观看后会写入观看记录，点赞、收藏、评论、弹幕、分享和稍后再看等互动会更新对应的数据表；创作者可以上传视频和封面，并在发布时使用 AI 生成文案、推荐分类和封面；管理员可以在后台查看请求日志和接口耗时。

在 API 设计上，所有业务接口统一使用 `/api/v1` 前缀，并尽量使用资源名表达 URI。比如发布视频使用 `POST /api/v1/videos`，记录观看使用 `POST /api/v1/videos/{id}/views`，查询我的视频使用 `GET /api/v1/users/me/videos`。我们使用 GET、POST、PUT、DELETE 表达不同操作，并用 400、401、403、404、500 等状态码区分错误类型。

数据设计方面，项目包含用户、视频、点赞、收藏、观看历史、评论、弹幕、稍后再看、关注关系、聊天消息、分享记录和请求日志等核心表。权限方面，除注册和登录外，核心接口都需要 JWT；删除视频时必须校验当前用户是否是作者；我的视频、观看历史和通知都只能访问当前用户自己的数据。

项目的扩展亮点是 AI 辅助模块，包括视频文案生成、视频分类推荐、视频封面推荐和评论回复建议。AI 模块依赖外部服务配置，如果服务未配置或调用失败，会返回明确错误，但不会阻断基础发布流程，用户仍然可以手动填写内容。

团队协作上，我们采用低耦合功能负责人制。组长负责公共能力、鉴权、数据库、集成和文档；其他成员分别负责推荐流、互动通知、视频发布、删除权限和日志监控等模块。每位成员负责自己模块的接口说明、自测和异常场景，最后由组长统一完成集成冒烟测试。

总结来说，我们的项目完成了短视频社区从浏览、互动、发布到监控的核心业务闭环，同时把 RESTful API 规范、JWT 鉴权、数据库设计、权限控制和日志监控落到了实际代码中。以上就是我们的项目汇报，谢谢老师。
