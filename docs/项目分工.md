# API 课程期末作业项目分工

## 一、分工原则

本项目为 API 课程期末作业，评分重点不仅包括功能完成度，也包括 API 设计规范、前后端联调质量、鉴权、日志、监控和数据存储设计。因此本项目采用“低耦合功能负责人制”：

1. 每个功能模块的前端页面、后端接口、接口联调和自测由同一位成员负责。
2. 每位成员优先负责独立页面、独立接口和独立数据逻辑，尽量避免多人同时修改同一个 Controller 方法。
3. 公共能力由组长统一维护，包括项目结构、统一请求封装、鉴权、最终集成和文档汇总协调。
4. 所有接口必须遵守 RESTful API 设计规范，统一使用 `/api/v1` 前缀。
5. 所有需要登录的接口必须携带 `Authorization: Bearer <token>`。
6. 每位成员负责自己功能接口的 API 规范性审核，包括 URI、HTTP 方法、状态码、请求参数、响应字段和响应格式。
7. API 规范具体要求以 `docs/API_DESIGN_GUIDE.md` 为准。
8. API 文档采用共享文档协作方式，每位成员负责填写自己模块的接口说明。
9. 每位成员完成自己模块后，需要补充或更新对应接口说明、测试数据和验收截图。
10. 每位成员负责自己模块的功能自测、接口自测和异常场景自测。
11. 组长负责项目初始化、公共模块搭建、共享 API 文档模板协调、最终集成冒烟测试、文档整合和提交验收。

## 三、并行开发边界

为了保证 5 位成员可以同时开展工作，开发时按以下边界推进：

| 成员 | 主要修改范围 | 尽量避免修改 |
| --- | --- | --- |
| 陈茹（组长） | 项目配置、数据库初始化脚本、基础表结构、公共工具、鉴权拦截、首页视频播放基础组件、项目部署配置、共享 API 文档模板、最终集成 | 各成员已经完成的业务逻辑细节 |
| 组员 A | 推荐流页面、推荐接口、观看记录、切换逻辑 | 点赞数据写入、上传存储、日志统计 |
| 组员 B | 点赞按钮、点赞接口、点赞记录表、点赞状态字段 | 推荐排序逻辑、视频上传、删除权限 |
| 组员 C | 发布页面、上传接口、文件访问路径、我的视频分页列表 | 删除视频权限、日志监控、推荐去重 |
| 组员 D | 删除按钮、删除接口权限校验、请求日志、耗时统计、验收截图 | 推荐算法、点赞规则、上传存储 |

如果两个模块需要共享字段或接口，先在共享 API 文档中约定字段，再分别实现。例如推荐流需要展示 `liked` 和 `likeCount` 时，组员 B 负责点赞状态和点赞数的计算规则，组员 A 负责在推荐流返回中按约定字段展示和使用。涉及自己模块的 URI、HTTP 方法、状态码和响应格式，由对应负责人按照 `docs/API_DESIGN_GUIDE.md` 自查。

数据库由组长统一初始化并建立基础表结构。其他成员如果需要新增字段、索引或表，需要先在共享文档中说明用途和字段含义，再由组长统一合并到数据库说明或建表脚本中，避免多人同时修改表结构导致冲突。

- 该成员主要负责安全边界和监控验收，尽量不改推荐、点赞、上传的核心业务逻辑。

## 五、API 设计规范要求

所有成员开发接口时必须遵守以下规范，避免课程验收时因 API 风格不统一扣分。具体细节以 `docs/API_DESIGN_GUIDE.md` 为准，每位成员负责自己功能接口的规范性审核和响应格式检查。

### 1. URI 规范

| 要求 | 示例 |
| --- | --- |
| 使用资源名，不使用动作名 | `/api/v1/videos` |
| 使用复数资源名 | `/api/v1/users/me/videos` |
| 使用统一版本前缀 | `/api/v1` |
| 子资源使用层级表达 | `/api/v1/videos/{id}/views` |
| 查询、分页使用 query string | `/api/v1/users/me/videos?page=1&limit=6` |

### 2. HTTP 方法规范

| 方法 | 使用场景 | 示例 |
| --- | --- | --- |
| `GET` | 查询资源 | `GET /api/v1/videos/recommendations` |
| `POST` | 创建资源 | `POST /api/v1/videos` |
| `PUT` | 更新状态 | `PUT /api/v1/videos/{id}/like` |
| `DELETE` | 删除资源 | `DELETE /api/v1/videos/{id}` |

### 3. 状态码规范

| 状态码 | 场景 |
| --- | --- |
| `200 OK` | 查询、更新、删除成功 |
| `201 Created` | 注册、发布视频成功 |
| `400 Bad Request` | 参数错误 |
| `401 Unauthorized` | 未登录或 Token 无效 |
| `403 Forbidden` | 没有权限 |
| `404 Not Found` | 资源不存在 |
| `500 Internal Server Error` | 服务端异常 |

### 4. 响应体规范

统一返回基础结构：

```json
{
  "success": true,
  "message": "操作成功",
  "data": {}
}
```

错误响应示例：

```json
{
  "success": false,
  "message": "无权限删除该视频"
}
```

## 六、协作流程

1. 每位成员先确认自己负责的接口路径、请求参数和响应字段。
2. 后端接口完成后，使用 Postman、Apifox 或浏览器控制台完成自测。
3. 前端页面接入自己负责的接口，保证页面流程能完整跑通。
4. 每个功能提交前，需要至少覆盖成功、失败、未登录或越权中的一种异常场景。
5. 每位成员准备自己模块的测试截图、接口调用结果和异常场景说明。
6. 每位成员按照 `docs/API_DESIGN_GUIDE.md` 自查自己接口的 API 规范和响应格式。
7. API 共享文档由组长提供模板，每位成员各自填写自己负责的接口说明。
8. 如果需要修改别人负责的接口或字段，先在组内同步，由组长确认后再改。
9. 每天同步当前进度、已完成接口、遇到的问题和下一步计划。
10. 最终由组长汇总检查共享 API 文档、鉴权、日志和整体演示流程完整性，并完成集成冒烟测试。
